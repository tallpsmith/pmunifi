# Research: UniFi PCP PMDA

**Feature Branch**: `001-unifi-pmda` | **Date**: 2026-03-15

## R-001: PCP Python PMDA Conventions

**Decision**: Follow upstream PCP Python PMDA patterns (gluster, openmetrics, simple) exactly.

**Rationale**: PCP has a well-established convention for Python PMDAs. Deviating invites breakage with PMCD and confuses PCP-literate operators.

**Alternatives considered**: None — constitution principle II ("PCP Conventions First") makes this non-negotiable.

### Key Findings

**File layout** in `$PCP_PMDAS_DIR/unifi/`:

| File | Purpose |
|---|---|
| `pmda_unifi.python` | Main PMDA script (shebang `#!/usr/bin/env pmpython`) |
| `Install` | Shell script sourcing `pmdaproc.sh` to register with PMCD |
| `Remove` | Shell script to deregister |
| `domain.h` | Auto-generated by framework via `PCP_PYTHON_DOMAIN` env var |
| `pmns` | Auto-generated by framework via `PCP_PYTHON_PMNS` env var |
| `help` | Optional; help text can be provided inline via `add_metric(..., oneline=, text=)` |
| `unifi.conf` | INI-style configuration file |

**Install script pattern** (from pmdagluster):
```sh
#!/bin/sh
. $PCP_DIR/etc/pcp.env
. $PCP_SHARE_DIR/lib/pmdaproc.sh
iam=unifi
python_opt=true
daemon_opt=false
pmdaSetup
pmdaInstall
exit
```

Custom interactive prompting goes *before* `pmdaSetup`/`pmdaInstall`.

**PMDA class hierarchy**: `PMDA` extends `MetricDispatch`. Key methods:
- `add_metric(name, pmdaMetric(...), oneline=, text=)` — register metrics
- `add_indom(pmdaIndom(serial, dict_or_list))` — register instance domains
- `set_fetch_callback(fn)` — per-metric/instance fetch (signature: `fn(cluster, item, inst) → [value, status]`)
- `set_fetch(fn)` — once-per-PDU pre-fetch hook (good for snapshot swap)
- `set_label(fn)` / `set_label_callback(fn)` — domain/indom/item and per-instance labels
- `set_user(username)` — drop privileges
- `replace_indom(serial, dict)` — update dynamic instance domains
- `inst_lookup(indom, inst)` — retrieve cached ctypes.Structure in fetch callback
- `run()` — enter PMCD dispatch loop (checks env vars for mode: probe/domain/pmns/dispatch)

**Instance domains — dict-based (pmdaCache)**: Pass a `dict` with string keys (instance names) and `ctypes.Structure` values to `pmdaIndom`. The cache assigns stable internal instance IDs persisted to disk. This is mandatory for archive continuity when devices appear/disappear.

**Threading rule**: Only the main thread (running `cpmda.pmda_dispatch()`) may call PMDA library functions. Background threads collect data; main thread serves it. Python's GIL makes reference assignment (`self._snapshot = new_snapshot`) atomic — no locks needed on the read path for copy-on-write.

**Metric semantics mapping**:
- `PM_SEM_COUNTER` — monotonically increasing (rx_bytes, tx_bytes, packets, errors)
- `PM_SEM_INSTANT` — point-in-time snapshots (client count, PoE power)
- `PM_SEM_DISCRETE` — rarely changing metadata (model, firmware, port speed)

## R-002: UniFi Controller REST API

**Decision**: Use legacy `/api/s/{site}/stat/*` endpoints with API key auth (`X-API-Key` header). The official `/integration/v1/` endpoints lack granular per-port counters.

**Rationale**: The `/stat/device` endpoint returns the full `port_table` with per-port traffic counters, PoE fields, uplink data, and SFP stats. The official API is high-level device CRUD only.

**Alternatives considered**: Official `/integration/v1/` API — rejected because it doesn't expose per-port counters, DPI data, or health subsystems. May revisit when the official API matures.

### Endpoints Required

| Endpoint | Method | Purpose |
|---|---|---|
| `/api/self/sites` or `/api/stat/sites` | GET | Site discovery (with health/counts) |
| `/api/s/{site}/stat/device` | GET | Full device stats including `port_table`, `uplink`, `radio_table` |
| `/api/s/{site}/stat/sta` | GET | Connected client list with switch port mappings |
| `/api/s/{site}/stat/health` | GET | Site health subsystems (wan, wlan, lan, vpn) |
| `/api/s/{site}/stat/sitedpi` | POST | DPI category stats (type=by_cat) |

### UDM Path Prefix

UniFi OS devices (UDM, UDR, UCG) require `/proxy/network` prefix before `/api/...`. Standalone controllers do not. Config option `is_udm` (FR-018) controls this.

### Response Envelope

All legacy endpoints return: `{"meta": {"rc": "ok"}, "data": [...]}`. Check `meta.rc == "ok"` and iterate `data` array.

### Key Port Table Fields

**Traffic counters**: `rx_bytes`, `tx_bytes`, `rx_packets`, `tx_packets`, `rx_errors`, `tx_errors`, `rx_dropped`, `tx_dropped`, `rx_broadcast`, `tx_broadcast`, `rx_multicast`, `tx_multicast`

**PoE**: `port_poe`, `poe_enable`, `poe_good`, `poe_power`, `poe_voltage`, `poe_current`, `poe_mode`, `poe_class`

**Status**: `port_idx`, `name`, `up`, `enable`, `speed`, `full_duplex`, `is_uplink`, `media`, `stp_state`, `satisfaction`

**Pre-computed rates** (excluded per FR-020): `rx_bytes-r`, `tx_bytes-r`, `bytes-r`

### Topology Discovery

No explicit LLDP table endpoint. Topology is derived from:
- Each device's `uplink` field (upstream device MAC + port index)
- Client `sw_mac` + `sw_port` fields (which switch port a client connects to)
- `is_uplink` flag on port_table entries

### Polling Considerations

- No documented rate limits on data query endpoints
- 30s poll interval is the community standard (unpoller default)
- API key auth avoids session expiry / login rate limiting entirely
- Minimum 10s interval recommended for constrained devices

## R-003: Python Project Structure

**Decision**: `src/` layout with two-phase deployment (pip install → setup command deploys to `$PCP_PMDAS_DIR`).

**Rationale**: `src/` layout prevents import shadowing and forces install discipline — critical for a package with a separate deployment target. The two-phase approach is the only practical way to bridge pip's site-packages with PCP's `$PCP_PMDAS_DIR`.

**Alternatives considered**: Flat layout — rejected because tests could accidentally import from the working directory instead of the installed package. RPM-only distribution — rejected because it limits portability (FR-022 requires PyPI).

### Package Name

- PyPI: `pcp-pmda-unifi`
- Import: `pcp_pmda_unifi`

### PCP System Dependency

`pcp` Python bindings are NOT on PyPI. Must be installed via system package manager (`pcp-libs-python` on RPM, `python3-pcp` on Debian). Fail fast at import time with a helpful error message.

### Entry Points

```toml
[project.scripts]
unifi2dot = "pcp_pmda_unifi.cli:main"
pcp-pmda-unifi-setup = "pcp_pmda_unifi.setup:main"
```

### Testing

pytest with three tiers via markers:
- `unit` — fast, isolated, no PCP, no network (~70%)
- `integration` — mocked PCP and mock HTTP server (~20%)
- `e2e` — requires real PCP installation (~10%)

`--import-mode=importlib` with `src/` layout per pytest docs.

## R-004: Domain Number Selection

**Decision**: Use domain number 165 (unregistered, available in PCP's domain range).

**Rationale**: PCP reserves domains 0–127 for internal use and well-known PMDAs. Third-party PMDAs should use 128–511. Need to verify 165 is not claimed by checking `$PCP_VAR_DIR/pmns/stdpmid`.

**Alternatives considered**: Using a very high number (400+) to avoid collisions — unnecessary if we check the registry.

## R-005: Configuration File Format

**Decision**: INI-style config with `[controller:NAME]` sections, parsed by Python `configparser`.

**Rationale**: INI is the dominant config format for PCP PMDAs. `configparser` is stdlib (no extra deps). Named controller sections support multi-controller (FR-016) cleanly.

**Alternatives considered**: TOML — requires `tomllib` (Python 3.11+) or `tomli` (extra dep). YAML — extra dep. JSON — hostile to humans.

### Config Structure

```ini
[global]
poll_interval = 30
max_clients = 1000
grace_period = 300
enable_dpi = false

[controller:main]
url = https://192.168.1.1
api_key = your-api-key-here
sites = all
is_udm = true
verify_ssl = true
# ca_cert = /path/to/ca-bundle.crt
```
